Act as a Staff Distributed Systems Architect.

Decision context:
[REQUIRED: What problem is being solved]
[REQUIRED: The constraints the solution must satisfy]
[REQUIRED: The evaluation criteria — what matters most]

Options to evaluate:
Option A: [Name and description]
Option B: [Name and description]
Option C: [Name and description — add or remove as needed]

Evaluation dimensions:
[REQUIRED: The specific dimensions relevant to this decision.
Examples:
- Consistency guarantees
- Operational complexity
- Team expertise alignment
- Scalability path to [target]
- Migration cost from current state
- Vendor lock-in risk]

Task:
Evaluate all options against every stated dimension.
For each option-dimension combination: provide a specific assessment,
not a general preference.
Explicitly state where the evaluation depends on organizational context
rather than objective technical merit.
Identify which option best satisfies which constraint — there may not
be one option that dominates all dimensions.

Output:
Tabulate the evaluation: Option | [Dimension 1] | [Dimension 2] | ... | Overall.
Use ratings: Strong | Adequate | Weak — with one-sentence justification per cell.
Follow the table with a recommendation section (3 paragraphs maximum):
- Paragraph 1: recommended option and the primary technical justification
- Paragraph 2: the most significant trade-off accepted by this choice
  and the organizational condition under which it is acceptable
- Paragraph 3: the specific condition or metric that should trigger
  a reconsideration of this choice
Do not hedge the recommendation — commit to one option with explicit reasoning.
State any dimensions where the options are genuinely equivalent,
so stakeholder attention focuses where differentiation is real.

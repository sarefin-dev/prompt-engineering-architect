Act as a Principal Architect evaluating a build vs. buy decision.

Decision context:
[REQUIRED: The capability being evaluated — what functionality is needed]
[REQUIRED: The build option — build in-house or extend an open-source project]
[REQUIRED: The buy options — vendor products or SaaS solutions]
[REQUIRED: Strategic context — is this capability core to the business, or commodity?]

Evaluation dimensions:
[REQUIRED: List the dimensions relevant to this decision]
Suggested dimensions:
- Time to capability (when is the capability needed?)
- Total cost of ownership over 3 years
- Customization requirements (standard capability vs. highly specific)
- Vendor risk (lock-in, vendor stability, data sovereignty)
- Team capability to build and maintain
- Integration complexity

Task:
Evaluate the build vs. buy options against the stated dimensions.

Apply the strategic filter first:
Is this capability a core competitive differentiator?
  If yes: building in-house creates strategic advantage. Evaluate build seriously.
  If no: commodity capabilities should be bought unless build has a specific advantage.

For the leading options (one build, one or two buy):
Evaluate against every stated dimension.
Calculate the 3-year TCO for each option.

Output:
State the strategic classification — core or commodity — and its implication.
Tabulate: Option | [Dimension 1] | ... | 3-year TCO | Recommendation.
Provide the recommendation with explicit build/buy rationale.
State the vendor risk assessment for each buy option:
  - What is the lock-in cost if the vendor changes pricing or is acquired?
  - Is the data portable if you need to migrate?
  - What is the SLA and support model?
State the build risk assessment:
  - What is the team's credible ability to build and maintain this?
  - What is the opportunity cost — what else could the team build instead?
Do not recommend building commodity capabilities to avoid vendor dependency
without a credible plan for the ongoing maintenance burden.
